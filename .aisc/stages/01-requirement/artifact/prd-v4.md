## 1. 功能目标
建立一个基于 AI 主题的视频分享平台，允许注册用户上传、发现、浏览和收藏 AI 相关视频内容，构建一个连接 AI 爱好者和创作者的社区。

## 2. 用户故事
| 角色 | 行为 | 期望结果 |
|------|------|---------|
| 注册用户 | 上传一个新的 AI 视频 | 视频成功上传，并展示在视频列表中，标题、描述和标签信息准确。 |
| 注册用户 | 在视频列表中通过关键词搜索 | 能够快速找到标题、描述或标签中包含该关键词的视频。 |
| 注册用户 | 浏览视频列表 | 能够看到按发布时间（默认）或热度排序的视频，并可以翻页查看。 |
| 注册用户 | 收藏一个喜欢的视频 | 该视频被添加到我的收藏列表中，并且收藏状态实时更新（如心形图标变红）。 |
| 注册用户 | 查看我的收藏列表 | 能够看到所有我收藏过的视频列表，并可以取消收藏。 |
| 未登录用户 | 访问平台 | 首页可浏览视频列表，但收藏按钮需提示登录或跳转登录页；直接访问上传/收藏页面会被重定向到登录页，且登录后携带redirect参数回跳。 |

## 3. 功能点
### 3.1 用户注册与登录
- **描述**：提供注册新用户和登录现有用户的功能。所有核心操作（上传、收藏）均需要身份认证。
- **输入**：
    - 注册：用户名、邮箱、密码
    - 登录：邮箱、密码
- **输出**：
    - 成功：跳转到视频列表首页或携带的redirect页面，生成并返回 JWT 令牌，并通过 Authorization Bearer 传递。
    - 失败：展示具体错误信息（如“邮箱已注册”、“密码错误”）。
- **令牌机制**：
    - 类型：JWT
    - 过期时间：24小时（Access Token），Refresh Token 有效期 7 天
    - 刷新：Access Token 过期后，前端使用 Refresh Token 通过 `/api/auth/refresh` 获取新 Access Token
    - 防范措施：
        - **CSRF 防护**：采用 HttpOnly Cookie + 响应头方案。服务端在登录成功后生成随机 CSRF Token，并通过 `Set-Cookie` 设置 HttpOnly、SameSite=Strict 的 Cookie（如 `csrf_token`），同时通过响应头 `X-CSRF-Token` 返回同一个 Token。前端将响应头中的 Token 存储在内存变量中，后续所有非 Get 请求（除登录/注册外）的请求头中添加 `X-CSRF-Token` 字段。服务端验证请求头中的 Token 是否与 Cookie 中的值一致。
        - **XSS 防护**：对所有用户输出进行 HTML 转义，禁止直接渲染未转义的内容。
- **前置条件**：无
- **后置条件**：用户的状态变为"已登录"，系统记录用户会话。

### 3.2 视频上传
- **描述**：允许已登录用户上传视频文件，并填写元数据。
- **输入**：
    - 视频文件（支持 mp4, webm, mov 格式，暂不支持 avi）
    - 标题（必填，最长 100 字符）
    - 描述（选填，最长 500 字符）
    - 标签（选填，最多 5 个，用逗号分隔，同一视频内去重，且自动转小写并过滤空格）
- **分片上传流程**：
    1. 前端将视频文件分片（每片 5MB，最后一片可小于 5MB）。
    2. 调用 `POST /api/upload/init` 创建上传任务，返回 `taskId` 及初始上传凭证。
    3. **并发控制**：前端应控制同时上传的分片数量不超过 3 个（避免浏览器并发限制与服务器压力）。使用队列调度，每次最多发起 3 个分片的上传，前一批次完成后再发起下一批。
    4. 对每个分片，先调用 `POST /api/upload/{taskId}/signPart` 获取该分片的临时上传 URL 及签名（有效期 15 分钟），然后将分片直接上传至对象存储。
    5. 上传完成后（所有分片均已成功上传），调用 `POST /api/upload/{taskId}/complete` 触发服务端合并文件并启动转码处理。
    6. 续传：若网络中断，续传时可先调用 `GET /api/upload/{taskId}/parts` 查询已成功上传的分片列表，前端跳过已上传分片，只上传缺失分片，并注意重新获取凭证（每个分片上传前获取新凭证）。
    7. 凭证刷新策略：若上传过程中凭证过期，自动重新请求该分片对应 `signPart` 并重试当前分片（最多重试 3 次），超过 3 次则标记该分片上传失败。
    8. 重试策略：分片上传失败（非凭证过期）时，自动重试 3 次，间隔 1/3/5 秒；仍失败则标记该分片上传失败，整体任务进入"上传失败"状态。
- **存储方案**：采用云存储直传方式，分片直接上传至对象存储（如阿里云OSS/腾讯云COS），服务端只负责签发和验证凭证，不承担数据代理转发。视频播放时通过 CDN 分发。
- **状态查询与推送**：
    - 查询接口：`GET /api/upload/{taskId}/status` 返回当前状态及进度信息（已完成分片数/总分片数）。
    - 推送机制：
        - **WebSocket 连接**：客户端在初始化上传任务后，向 `/ws/upload/{taskId}` 发起 WebSocket 连接，携带 JWT Token 进行鉴权（可通过 URL 参数或协议头）。
        - **心跳与重连**：服务端每 30 秒发送心跳帧，客户端若 10 秒内未收到则主动重连，重连间隔递增（1 秒、2 秒、4 秒…最多 30 秒）。
        - **事件类型**：服务端推送 `progress`（进度变化）、`status_change`（状态转换，如上传中→处理中→已发布/失败）、`error`（错误详情）。
        - **降级**：若 WebSocket 不可用（如网络环境限制），客户端降级为 5 秒轮询 `GET /api/upload/{taskId}/status`。
    - 状态流转：上传初始化后状态为"上传中" → 所有分片上传递交后进入"处理中"（服务端转码、截图）→ 处理完成后自动变为"已发布"或"失败"。转码失败后重试最多 2 次，若仍失败则标记为"上传失败"。
- **处理超时上限**：
    - 单文件上传总时长：从 init 到 complete 完成，超过 2 小时则自动标记为"上传失败"。
    - 转码阶段：单视频转码超时设为 30 分钟，超时后标记为"上传失败"。
- **输出**：
    - 成功：返回视频ID、最终状态（已发布/失败）。
    - 失败：返回具体错误（如"文件大小超限（最大500MB）"、"格式不支持"、"上传超时"等）。
- **前置条件**：用户已登录。
- **后置条件**：视频记录被创建，系统后台开始处理。视频出现在"我的上传"列表和公开视频列表中。
- **UI 状态**：
    - 上传中：显示进度条（百分比）、当前分片序号、预估剩余时间。
    - 处理中：显示"处理中…"动画。
    - 失败：显示失败原因及重试按钮。

### 3.3 视频列表浏览与搜索
- **描述**：展示所有已发布的视频，支持按关键词搜索，支持排序（默认按发布时间倒序，可切换为热度排序），支持分页。
- **输入**：
    - 搜索关键词（可选，提交式触发：用户点击搜索按钮或按回车后触发搜索；**无防抖**；服务端对关键词进行安全校验：去除首尾空格、限制长度 ≤100 字符、过滤 SQL 注入关键字、HTML 转义）
    - 页码（必填，默认为 1，若超出总页数则返回最后一页内容，前端显示最大页码，并在页面上提示"已到最后一页"）
    - 每页数量（可选，默认 20）
    - 排序方式（可选，默认 `created_at_desc`，支持 `hot_desc`）
- **输出**：
    - 成功：返回对应页的视频列表（包含标题、封面URL、作者、发布时间、播放量、收藏数、标签），以及总页数、当前页码、总视频数。
    - 空搜索：关键词无匹配时返回空列表，前端显示提示"未找到相关视频，请尝试其他关键词"。
    - 失败（如服务器错误）：返回错误信息，前端显示全局错误并显示重试按钮。
- **排序规则**：
    - 热度排序算法：`hot_score = 播放量 + 收藏数 × 3 + 时间衰减因子`，时间衰减因子定义：每过一天衰减 5%，即 `score = 原始分 × exp(-0.05*days)`，实现方案可使用定时任务或数据库函数。
- **服务端标签校验规则**：创建/更新视频时，服务端校验每个标签：长度 1~20 字符，只允许字母、数字、中文、下划线、减号，最多 5 个；若有非法标签则返回错误 "标签格式不正确"。
- **分页超限**：若请求的页码大于最大页数，服务端返回最后一页的数据，并在响应中标记 `isLastPage: true`，前端将页码显示为最大页码并提示"已到最后一页"。
- **响应式布局断点**：
    - 移动端：<576px（单列列表，搜索栏折叠）
    - 平板：≥576px 且 <768px（两列网格）
    - 桌面：≥768px 且 <992px（三列网格）
    - 大屏：≥992px（四列网格）
    - 具体实现可通过 CSS 媒体查询或前端框架 Grid 系统。

### 3.4 收藏功能（Toggle）
- **描述**：已登录用户可收藏或取消收藏一个视频，采用 toggle 接口实现原子操作。
- **输入**：
    - `POST /api/videos/{videoId}/favorite` （请求体不需要额外字段，或可包含 mode: toggle，但不强制）
    - 无需请求体，仅凭当前登录用户和视频 ID 进行切换。
- **输出**：
    - 200：返回 `{ "favorited": true/false }`，表示当前操作后是否处于收藏状态。
- **实现**：
    - 服务端使用 `INSERT INTO favorites (user_id, video_id) VALUES (?, ?) ON DUPLICATE KEY UPDATE deleted = NOT deleted` 或类似逻辑实现幂等切换。若原本未收藏则插入/设置状态为已收藏；已收藏则删除/标记取消。
    - 保证并发安全（数据库唯一约束或锁机制）。
- **前置条件**：用户已登录。
- **后置条件**：视频收藏数更新，用户收藏列表变更。前端立即更新 UI 状态（心形图标变色/恢复）。
- **未登录用户交互**：点击收藏按钮时弹出模态框提示"请先登录"，点击"去登录"跳转到 `/login?redirect=当前页`，登录成功后回到当前页，但不自动执行收藏操作（用户需再次点击）。
- **服务端校验**：检查 video_id 是否存在且已发布（非删除状态），否则返回 404。
