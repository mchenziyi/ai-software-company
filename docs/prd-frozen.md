## 1. 功能目标
构建一个AI视频平台，核心目标是让用户能够安全、便捷地管理自己的视频内容，并发现其他用户上传的有趣视频。通过设置登录门槛，确保用户数据的私密性和安全性。该平台应具备清晰的内容上传、浏览、搜索、收藏、个人管理（上传记录与收藏）等基础功能，为后续引入AI驱动的视频推荐、标签生成、摘要等高级功能奠定基础。

## 3. 功能点
### 3.1 用户注册与登录
- 描述：用户可以通过用户名和密码进行注册和登录，所有核心功能都必须在登录后才能使用。
- 输入：用户名、密码、确认密码（注册时）。
- 输出：登录成功后跳转到首页（视频列表页）；注册成功后自动登录并跳转到首页；失败时返回错误提示。
- 前置条件：无。
- 后置条件：系统为该用户生成一个唯一的Session或Token，用于后续的身份验证。

### 3.2 用户上传视频
- 描述：已登录用户可以从本地选择一个视频文件，并填写标题、描述和标签（可选），然后提交上传。上传完成后系统自动截取视频第一帧作为封面图（后台处理），视频进入后台处理（转码、封面提取等）。若转码成功，视频状态变为 available；若转码失败，视频状态变为 rejected，系统需记录明确的失败原因（如“视频格式不兼容”、“转码超时”等），并将此原因返回给前端用于展示。
- 输入：视频文件（mp4, mov, avi等常见格式）、标题（字符串，必填）、描述（文本，可选）、标签（字符串，多个用逗号分隔，可选）。
- 输出：上传成功后，系统返回上传成功的提示，视频状态变为 `pending`。在前端，该视频会立即出现在用户的“我的上传”列表中。经过后台处理后，状态会变为 `available` 或 `rejected`，并附带失败原因（若转码失败）。其他用户在视频列表中只能看到状态为 `available` 的视频。
- 前置条件：用户已登录，文件大小和格式符合预设规则（最大2GB，支持分片上传或预签名URL）。
- 后置条件：系统启动后台任务（转码、封面截取），视频状态流转：
  - 转码成功：`pending` → `available`
  - 转码失败：`pending` → `rejected`，并记录错误原因（如错误码+描述）。前端在用户查看“我的上传”时可获取此原因。

### 3.3 浏览视频列表
- 描述：用户可以浏览平台上所有用户的已发布（available状态）的视频列表，支持分页显示。
- 输入：页码（page）、每页固定20条（size固定）。
- 输出：一个包含视频信息（标题、作者、封面图、简介摘要、标签、上传时间、收藏数等）的列表。列表按上传时间倒序排列。**不包含播放量字段，前端可根据设计显示“暂无”或不展示。**
- 前置条件：用户已登录。
- 后置条件：无。

### 3.4 搜索视频
- 描述：用户在视频列表页顶部的搜索框中输入关键词，系统对视频的“标题”和“标签”进行模糊匹配，返回匹配的结果列表。匹配时不区分大小写，关键词长度限制1-50个字符。触发方式为点击搜索按钮或按下回车键。
- 输入：搜索关键词（字符串，1-50字符）。
- 输出：一个包含所有匹配视频的列表，同样支持分页，按上传时间降序排列。如果无匹配结果，则显示对应的空状态提示。
- 前置条件：用户已登录。
- 后置条件：无。

### 3.5 收藏/取消收藏视频
- 描述：用户在浏览视频列表或视频详情页时，可以通过点击“收藏”或“取消收藏”按钮来管理自己的收藏。采用乐观更新策略：点击后立即更新按钮状态（例如从空心心变为实心心），同时向后端发送请求；若请求失败，则回滚按钮状态并弹出错误提示（如“收藏失败，请重试”）。
- 输入：目标视频的ID。
- 输出：按钮状态即时改变，后端记录操作，并更新视频的收藏数（成功时）。失败时恢复原状态并提示错误。
- 前置条件：用户已登录。
- 后置条件：用户对该视频的收藏状态更新（收藏或取消收藏）。

### 3.6 查看个人收藏列表
- 描述：用户可以进入一个专门的“我的收藏”页面，查看自己所有收藏过的视频。
- 输入：用户ID。
- 输出：一个视频列表，展示所有该用户收藏的视频信息，按收藏时间倒序排列。列表分页每页固定20条。
- 前置条件：用户已登录。
- 后置条件：无。

### 3.7 查看个人上传列表（新增）
- 描述：用户可以进入“我的上传”页面，查看自己上传的所有视频，包括处理中（pending）、已发布（available）、被拒绝（rejected）的视频。rejected 视频通常是转码失败导致，系统会显示具体的失败原因（如“转码超时”）。每个视频卡片需显示当前状态（如“处理中”、“已发布”、“转码失败”），并针对 rejected 视频提供“删除”和“重新上传”操作按钮。点击“删除”可删除该视频记录（包括关联数据）；点击“重新上传”会弹出文件选择器，用户选择新文件后提交，系统将替换原记录（新建一个上传任务并删除原 rejected 记录），或采用其他合理方式（如直接覆盖上传）。具体流程由后端和前端协商，但必须满足用户能够重新上传同一视频的需求。
- 输入：用户ID。
- 输出：一个视频列表，展示所有该用户上传的视频信息，按上传时间倒序排列。列表分页每页固定20条。
- 前置条件：用户已登录。
- 后置条件：无。

## 4. 业务规则
- **认证规则**：所有核心业务功能（上传、浏览、搜索、收藏、查看收藏、查看上传）均要求用户必须处于登录状态。
- **上传规则**：
  - 单个视频文件大小上限：2GB。要求后端支持分片上传或预签名URL方式确保大文件可靠传输。
  - 支持的视频格式：.mp4, .mov, .avi, .mkv, .webm。
  - 视频标题长度限制：1-100个字符。
  - 视频描述长度限制：0-500个字符。
  - 标签数量限制：最多5个标签，每个标签长度限制1-20个字符。
- **视频状态与可见性**：
  - 状态枚举：`pending`（处理中）、`available`（已发布）、`rejected`（已拒绝）。
  - 状态流转：
    - 上传后 → `pending`
    - 转码成功 → `available`
    - 转码失败 → `rejected`，并记录错误原因（如“视频格式不兼容”、“转码超时”等）。
  - 可见性规则：
    - 普通用户（非上传者）只能看到 `available` 状态的视频。
    - 上传者可以在“我的上传”中看到自己所有状态的视频（包括 `pending`、`available`、`rejected`），并查看 rejected 视频的错误原因。
- **收藏规则**：
  - 一个用户对同一个视频只能收藏一次；重复收藏时提示“您已收藏过该视频”（前端通过状态判断，不发送请求）。
  - 用户收藏总数无上限。
  - 收藏/取消收藏采用乐观更新：先改变UI状态，再发请求；失败时回滚UI并显示错误提示。
- **搜索规则**：
  - 仅支持对视频“标题”和“标签”字段进行模糊匹配搜索，不匹配视频描述。
  - 不区分大小写。
  - 关键词长度限制：1-50个字符。
  - 触发方式：点击搜索按钮或按下回车键。
  - 搜索结果按上传时间降序排列。
- **内容审核**：所有上传的视频在 `available` 之前默认处于 `pending` 状态。本版本不涉及人工审核，但会通过后台转码处理自动判定通过或拒绝。转码失败即为 rejected。
- **分页规则**：所有列表（首页、搜索结果、收藏列表、个人上传列表）每页固定显示20个视频，客户端不允许通过参数自定义size。后端提供page（从1开始）参数，size固定为20。
- **排序规则**：平台视频列表默认按上传时间降序排列；个人收藏列表按收藏时间倒序排列；个人上传列表按上传时间降序排列。
- **错误处理**：对于转码失败的视频，系统需记录明确的错误原因（如错误码和描述），并在前端“我的上传”页面中展示该原因，方便用户了解失败详情并采取后续操作（删除或重新上传）。
- **转码失败后的用户操作**：用户可在“我的上传”页面中对 rejected 视频执行以下操作：
  - 删除：删除该视频及其所有关联数据（封面、标签、收藏等）。
  - 重新上传：用户选择新视频文件后提交，系统将创建一个新的上传任务并自动删除原 rejected 视频记录，新视频重新进入 pending 处理流程。

## 5. 非功能性需求（可选）
- **安全性**：所有用户文本字段（标题、描述、标签）在后端输出时进行HTML转义，防止XSS攻击；搜索时保留原始输入用于匹配。
- **空数据处理**：后端返回空列表，前端根据场景显示对应文案（如“暂无视频”、“您还没有收藏视频”）。
- **性能**：大文件上传需支持分片或预签名URL，避免单次请求超时。
- **可扩展性**：视频状态枚举预留 future 扩展（如审核中、已删除等），但本版本仅使用 pending、available、rejected。

## 6. 边界条件与极限值
- **文件大小**：上传视频文件大小最大2GB，超过时前端提前校验并提示。
- **标题长度**：1-100字符，超出或空时拒绝上传。
- **描述长度**：0-500字符，超出时拒绝上传。
- **标签数量**：最多5个，每个标签1-20字符，超出时拒绝上传。
- **搜索关键词长度**：1-50字符，少于1或超过50时提示限制。
- **分页尺寸**：固定20条，客户端不得自定义。
- **页码下限**：page最小为1，小于1时后端返回第一页。
- **转码失败原因**：错误原因字符串长度不超过200字符，由后端定义。

## 7. 依赖与假设
- **依赖**：后台转码服务必须能够返回明确的错误码和原因。
- **假设**：本版本中，所有转码失败均为技术原因（非内容违规），用户可自由选择删除或重新上传。
- **假设**：重新上传操作会彻底替换原失败视频，原视频的收藏记录将被删除（收藏影不影响其他用户？由于原视频已失败且被替换，原收藏应不再有效，建议一并清理。此细节可在后续设计确认。）
